# パフォーマンス設計原則

CoffeeBook Reader で起きた性能劣化を、他のアプリにも使える汎用的な設計メモとして残す。

他の作業ディレクトリで同じ観点を確認する場合は、[パフォーマンスホットパス確認テンプレ](PERFORMANCE_HOT_PATH_AUDIT_TEMPLATE.md) を使う。

## 事象の型

頻繁に通るUI遷移に、初回起動、明示同期、メンテナンスで行うべき重い処理を置くと、画面は正しくても体感上は使えなくなる。

今回の典型例:

- 一覧から本を開く。
- 一覧へ戻る。
- 別の本を開く。
- そのたびに全読み込み、ファイル走査、派生値再計算が走り、アプリが固まったように見える。

個々の処理は正しくても、置き場所が悪いと問題になる。読み込み、掃除、キャッシュ確認そのものが悪いのではなく、通常操作で何度も通るホットパスに置いたことが問題。

## 設計原則

初回読み込み後の永続データは、基本的にメモリ上の作業セットとして扱う。

- 永続データの全読み込みは、起動時、明示的な再読み込み、ライブラリ切り替えなどの境界で行う。
- 詳細画面から一覧へ戻るだけなら、ストレージを読み直さず、メモリ上の状態から表示を更新する。
- 変更はメモリ上に反映し、必要なタイミングで保存する。
- 不要ファイル掃除、全件スキャン、リモート同期確認は、起動時、明示操作、バックグラウンド処理、またはメンテナンスとして扱う。
- ファイル存在、ファイルサイズ、件数、容量サマリ、グループ化結果など、計算が重い派生値はキャッシュする。
- 派生値キャッシュは、元データやファイルが変わる操作で無効化する。
- 選択だけが変わる場合は、一覧全体を作り直さず、対象行や選択表示だけを更新する。

## ホットパス確認リスト

UIイベントへ処理を追加する前に確認する。

- そのイベントは、通常の移動、戻る、選択、スクロール、ページめくりで何度も発火するか。
- ディスク、ネットワーク、多数ファイル、画像デコード、JSONシリアライズ、大きなコレクションを触るか。
- 1件だけ変わったのに、一覧全体を再生成していないか。
- 起動時に済ませた処理を、画面復帰時に繰り返していないか。
- 結果をキャッシュし、変更操作側で無効化できないか。
- 明示操作、遅延実行、バッチ化、バックグラウンド化に逃がせないか。

該当する場合、その処理をUIホットパスで待たせない。

## CoffeeBook Reader での事象

症状:

- 本を開いて戻り、別の本を開くと固まったように遅い。
- 巻末から次巻へ移るまで時間がかかる。
- 本棚スクロールが重くなる。

主な原因:

- `BookshelfPage.OnAppearing` が読書画面から戻るたびにライブラリ全読み込みをしていた。
- 読書位置保存時に、ライブラリ保存だけでなく不要ファイル掃除まで待っていた。
- 本棚カードのキャッシュ表示で、ページ画像の存在確認とファイルサイズ計算を何度も繰り返していた。

改善:

- Reader本棚のライブラリ読み込みはページ初回だけにした。
- 読書画面から戻った時は、メモリ上の本リストを再表示するだけにした。
- 読書位置保存では毎回の不要ファイル掃除をしないようにした。
- 巻またぎ時の旧巻/新巻の二重保存を避けた。
- 本ごとのページキャッシュ統計をメモ化し、キャッシュ削除などの変更時に無効化するようにした。

## CoffeeMovie での確認

2026-06-22 の CoffeeMovie 確認では、Android Reader に CoffeeBook Reader と同じ型の問題が見つかった。Studio には「画面へ戻るたびに全ライブラリを再読み込みする」ほどの直撃はなかったが、選択、絞り込み、プレビュータイマーに軽〜中程度のホットパスがあった。

是正:

- `MovieLibraryStore` は初回ロード後の `library.json` をメモリ上の作業セットとして保持する。
- Reader 本棚は、プレイヤーから戻るだけでは全読み込みしない。ライブラリ更新があった時だけ再描画する。
- Reader のシリーズ/Season 開閉は、ストレージを読まずにメモリ上の行だけ組み直す。
- Studio の動画選択と字幕タグ検索は、ロード済みライブラリを参照する。
- Studio の動画棚サムネイルは、一覧再生成のたびに `BitmapImage` を作らずパスを渡す。
- Studio の別窓プレビューは、250ms タイマーごとの `File.Exists` を避け、動画パス変更時だけ確認する。
- Studio 終了時はプレビュータイマーを止め、MediaElement の再生と `Source` を解放する。

残る注意点:

- ライブラリ全体をメモリに持つため、メモリ使用量は動画数、字幕キュー数、学習メモ量に比例する。ただし放置時間に比例して増え続ける構造ではない。
- さらに大規模化した場合は、映画メタデータと字幕キュー/学習状態の分割保存を検討する。
- 画像デコード、字幕生成、Reader書き出し、Drive取り込みなどは重い処理なので、進捗表示、非同期実行、差分処理を前提にする。

Studioでも原則は同じ。通常の選択、タブ切り替え、スクロールはメモリ上のUI更新に留め、import / sync / export / cleanup / compression / capture のような明示操作だけが重い処理を開始する。

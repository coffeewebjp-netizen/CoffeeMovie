# CoffeeMovie update summary — 2026-07-14

This release consolidates the Studio playback fixes, PC/Android seek controls, CoffeeLearning registration visibility and Drive exchange, editable roundtrip export, and metadata-only Drive reflection.

## Studio playback and seeking

- Playback position is persisted per movie and restored when the preview source opens.
- Edit preview, full preview, and the popup use explicit play/pause intent instead of rebuilding healthy media on every resume.
- Slider thumb drags seek when the drag completes; slider-track clicks calculate the target time directly.
- Pending seeks temporarily own the slider so the periodic UI timer cannot restore a stale position.
- Stalled WPF media pipelines are detected and rebuilt with the requested position and play/pause state.
- Audio is suppressed during seek/recovery preroll and restored after the target is confirmed, preventing audible fragments from the start of a file.
- The popup is the sole preview audio source while open.
- Both PC preview surfaces provide `-1`, `-5`, `-custom`, `+1`, `+5`, and `+custom` controls. The custom interval is 1–9999 seconds; Left/Right uses it, Shift selects 5 seconds, and Ctrl selects 1 second.

## Android Reader controls

- Fullscreen playback provides symmetric one-second, five-second, and custom rewind/fast-forward controls.
- One saved custom interval, from 1 to 9999 seconds, is shared by both directions.
- Relative seeks are clamped to the media start/end and immediately refresh subtitle and saved-position state.

## CoffeeLearning registration state

- Studio and Reader display `CoffeeLearning登録済` beside the user memo and in subtitle overlays.
- Successful registrations create a device-specific Drive snapshot.
- Studio imports every available device snapshot from its configured Drive folder; Reader downloads and merges every snapshot through the Drive API.
- Registration merge is monotonic: a registered state is not cleared by an empty or older snapshot, and missing IDs can be filled by another device.
- AI notes and user notes are not overwritten by this registration-only exchange.

Known constraint: simultaneous pre-sync registration of the same cue on multiple devices can retain different CoffeeLearning word IDs or create a server duplicate. A server-side idempotency key based on movie, track, cue, and deck identity is still required for full duplicate prevention.

## Editable roundtrip exchange

- Studio can export one movie as `manifest.json`, subtitle SRT files, and UTF-8 BOM `notes.csv`.
- SRT files are authoritative for cue text/timing edits.
- `notes.csv` carries AI notes, user notes, tags, flags, and CoffeeLearning registration fields.
- Import preserves cue IDs where indexes match, keeping learning state attached after subtitle edits.

## Drive reflection contract

- `Driveへ全件反映` checks every movie without requiring manual selection.
- Matching fingerprints are skipped.
- Tag, note, thumbnail, subtitle, and other metadata-only changes rewrite only `.coffeemovie.json`.
- `.coffeemovie` package bytes are created or rewritten only when the package is missing or the video identity changed.
- The explicit single-movie `スマホ用書き出し` remains a full package export.
- Sidecar metadata is authoritative. Studio and Reader preserve a newer sidecar when extracting video bytes from an older package manifest.

This keeps Google Drive from uploading the full video again after ordinary tag or memo changes while preserving first-device and reinstall behavior.

## Verification

- Studio build succeeds with zero warnings and zero errors.
- Android Reader build succeeds with zero warnings and zero errors.
- Verification covers metadata-only package behavior: package bytes remain identical after a tag update, updated sidecar values win over an older package manifest, unchanged metadata is skipped, and a changed video identity rebuilds the package.

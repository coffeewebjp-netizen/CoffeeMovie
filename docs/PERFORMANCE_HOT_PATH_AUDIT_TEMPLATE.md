# パフォーマンスホットパス確認テンプレ

このテンプレは、Webアプリ、デスクトップアプリ、モバイルアプリで「通常操作のたびに重い処理が走っていないか」を確認するために使う。

基本思想は、初回または明示操作で全量読み込みし、その後の通常操作はメモリ上の作業セットと差分更新で済ませること。

## 使い方

対象リポジトリのルートで、以下を順に確認する。

- ユーザーが頻繁に通る操作を列挙する。
- その操作に紐づくイベントハンドラ、画面遷移、ライフサイクル処理を探す。
- そこでディスク、ネットワーク、画像デコード、JSON、DB、全件リスト再構築、同期、掃除が走っていないか確認する。
- 重い処理がある場合、初回だけ、明示操作だけ、バックグラウンド、差分更新、キャッシュ無効化方式に移せるか判断する。

## リスク表

見つけた箇所はこの表にまとめる。

| 重要度 | ファイル/行 | 操作 | 現在の処理 | 懸念 | 推奨対応 |
|---|---|---|---|---|---|
| High/Medium/Low |  |  |  |  |  |

重要度の目安:

- High: 戻る、選択、スクロール、ページ送りなどの通常操作でディスク/ネットワーク/全件再構築が走る。
- Medium: 保存やタブ切り替えなど比較的少ない操作だが、大規模データで固まる可能性がある。
- Low: 現状は問題ないが、件数増加時にキャッシュや差分化を検討したい。

## 改善パターン

- 初回ロード済みフラグを持ち、通常復帰では全読み込みしない。
- 一覧データはメモリ上の作業セットとして持ち、表示だけ再計算する。
- 選択状態だけの変更は、一覧全体ではなく対象行だけ更新する。
- ファイルサイズ、件数、容量サマリ、グループ化、ソートキーなどの派生値はキャッシュする。
- 派生値キャッシュは、元データやファイルを変更した操作で明示的に無効化する。
- 保存処理から掃除、同期、全件スキャンを分離する。
- 掃除、同期、整合性確認は起動時、明示操作、バックグラウンド、メンテナンスに寄せる。
- 画像ロード/デコード/サムネイル生成は非同期化し、表示領域に入ったものから遅延ロードする。
- 大量更新はプログレス表示、キャンセル、バッチ処理を付ける。
- 同期や書き出しは既存ファイル/メタデータ/ハッシュ/更新日時を見て、同一ならスキップする。

## Codex へ投げる依頼文

別の作業ディレクトリで同じ確認を依頼する時は、次を使う。

```text
この作業ディレクトリで、通常操作のホットパスに重い処理が入っていないか確認してください。

観点:
- 初回だけ全量読み込みし、通常の戻る/選択/スクロール/ページ送りではメモリ上の状態と差分更新で済んでいるか。
- 画面復帰、タブ切り替え、項目選択、スクロール、ページ送りで、ディスクI/O、ネットワーク、DB/JSON全読み込み、多数ファイル走査、画像デコード、一覧全体再生成、掃除、同期が走っていないか。
- 表示用getter/computed/propertyがファイル存在確認、ファイルサイズ計算、集計などの重い処理を繰り返していないか。
- 保存処理に不要ファイル掃除、全件スキャン、リモート同期確認などが混ざって通常操作を待たせていないか。
- 重い派生値はキャッシュされ、元データ変更時に無効化されているか。

出力:
- 同種の問題があるかの結論。
- High/Medium/Low のリスク表。
- すぐ直すべき箇所と、将来的な注意点。
- 可能なら改善方針を「初回ロード」「通常復帰」「差分更新」「キャッシュ」「無効化」「明示操作/バックグラウンド」に分けて提案してください。
```
